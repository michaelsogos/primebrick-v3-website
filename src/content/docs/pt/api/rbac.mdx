# RBAC — Role-Based Access Control

Primebrick implementa um modelo de controle de acesso baseado em roles (**RBAC**) com permissões
estruturadas segundo uma notação **wildcard**. A autorização é gerenciada em colaboração com
**Casdoor**, que mantém o mapeamento de roles entre usuários, roles e permissões.

## Permissões wildcard

As permissões no Primebrick seguem o formato **`modulo.acao.granularidade`**:

```text
modulo.acao.granularidade
  │       │        │
  │       │        └── nível de granularidade (opcional, pode ser *)
  │       └── ação específica (read, write, delete, ...)
  └── módulo ou recurso (users, documents, settings, ...)
```

### Exemplos

| Permissão | Significado |
|-----------|-------------|
| `users.read` | Pode ler os usuários |
| `users.read.*` | Pode ler os usuários a qualquer nível de granularidade |
| `users.write` | Pode modificar os usuários |
| `users.delete` | Pode excluir os usuários |
| `documents.read.own` | Pode ler apenas seus próprios documentos |
| `documents.read.department` | Pode ler os documentos do seu departamento |
| `documents.read.*` | Pode ler todos os documentos |
| `*.*.*` | Acesso completo (admin bypass) |

### Wildcards

O caractere `*` funciona como wildcard e pode aparecer em qualquer posição:

- `users.*` — todas as ações sobre o módulo `users`
- `*.read` — ação `read` sobre todos os módulos
- `*.*.*` ou `*` — acesso completo a tudo

### Granularidade

O terceiro nível (`granularidade`) define **o escopo** da permissão. Os valores comuns são:

| Granularidade | Descrição |
|---------------|-----------|
| `own` | Apenas os recursos criados pelo usuário |
| `department` | Os recursos do departamento do usuário |
| `organization` | Os recursos da organização do usuário |
| `*` | Todos os recursos, sem restrições |

Se a granularidade for omitida, assume-se `*` (sem restrições).

## Atribuição de roles com Casdoor

**Casdoor** gerencia o mapeamento entre usuários, roles e permissões. O fluxo é o seguinte:

1. Um **usuário** é atribuído a uma ou mais **roles** no Casdoor
2. Cada **role** tem um conjunto de **permissões** associadas
3. Quando o usuário se autentica, o JWT contém as roles atribuídas
4. O backend extrai as permissões das roles e verifica o acesso

### Estrutura no Casdoor

```text
Usuário: m.rossi@example.com
  └── Roles: [admin, editor]
        ├── admin → [*.*.*]
        └── editor → [documents.read.*, documents.write.own, users.read]
```

### Token JWT

O JWT emitido pelo Casdoor inclui as roles do usuário no claim correspondente. O backend
resolve as roles nas permissões correspondentes e as armazena no contexto da requisição
para as verificações de autorização.

## Verificação de permissões no backend

O backend verifica as permissões a cada requisição. A lógica de verificação compara a permissão
requerida pelo endpoint com as permissões efetivas do usuário:

```typescript
// Exemplo conceitual da verificação
function hasPermission(userPermissions: string[], required: string): boolean {
  return userPermissions.some(p => matchWildcard(p, required));
}

// Verifica se "users.read.*" cobre "users.read.own"
hasPermission(['users.read.*'], 'users.read.own'); // true

// Verifica se "*.read" cobre "documents.read"
hasPermission(['*.read'], 'documents.read'); // true
```

O matching de wildcards segue regras padrão: `*` em uma posição corresponde a
qualquer valor nessa posição.

## Admin bypass

Usuários com a role **admin** (ou com a permissão `*.*.*`) têm **acesso completo**
a todos os recursos e todas as ações, ignorando todas as verificações de autorização.

Isso é implementado como primeira verificação no middleware de autorização:

```text
SE o usuário tem a permissão *.*.*:
  → permite o acesso (bypass completo)
CASO CONTRÁRIO:
  → verifica a permissão específica requerida pelo endpoint
```

## Endpoints e permissões requeridas

Cada endpoint do backend declara a permissão requerida. Por exemplo:

| Endpoint | Permissão requerida |
|----------|---------------------|
| `GET /api/v1/users` | `users.read` |
| `POST /api/v1/users` | `users.write` |
| `DELETE /api/v1/users/:id` | `users.delete` |
| `GET /api/v1/ws/document-service/documents` | `documents.read` |

Se o usuário não tem a permissão requerida, o backend retorna um erro **403 Forbidden**
com um Problem Details que indica a permissão faltante (ver [Tratamento de erros](/pt/api/error-handling/)).
