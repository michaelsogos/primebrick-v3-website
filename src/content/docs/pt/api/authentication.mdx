# Autenticação

Primebrick utiliza **Casdoor** como Identity Provider (IdP) externo. A autenticação é baseada
no protocolo **OIDC** (OpenID Connect) e os tokens emitidos são **JWT** (JSON Web Token).

## Visão geral

Existem **três tipos de credenciais** para se autenticar na API do Primebrick:

| Tipo | Caso de uso | Header |
|------|-------------|--------|
| **Token de usuário** | Aplicações frontend (usuários reais) | `Authorization: Bearer <jwt>` |
| **Token de cliente OAuth2** | Comunicações service-to-service | `Authorization: Bearer <jwt>` |
| **Chave API** | Scripts, integrações, API Explorer | `X-API-Key: <api-key>` |

Os três mecanismos são validados pelo backend, que verifica a identidade e as permissões
associadas à credencial antes de processar a requisição.

## 1. Token de usuário (JWT)

O fluxo primário de autenticação para aplicações frontend é o **fluxo OIDC
Authorization Code**. O usuário é redirecionado ao Casdoor para fazer login, e recebe em troca
um JWT que é enviado a cada requisição à API.

### Fluxo

1. O frontend redireciona o usuário para a página de login do Casdoor
2. O usuário insere suas credenciais no Casdoor
3. O Casdoor redireciona ao frontend com um authorization code
4. O frontend troca o authorization code por um JWT (access token)
5. O frontend envia o JWT a cada requisição à API

### Uso

```bash
curl -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..." \
  http://localhost:3001/api/v1/users
```

O backend valida o JWT verificando:

- A **assinatura** utilizando as chaves públicas do Casdoor (JWKS)
- A **expiração** (claim `exp`)
- O **emissor** (claim `iss`) corresponde ao Casdoor
- A **audiência** (claim `aud`) corresponde à aplicação Primebrick

### Refresh token

Os JWT têm uma expiração limitada (tipicamente 1 hora). O frontend utiliza o refresh
token para obter um novo JWT sem exigir que o usuário faça login novamente.

## 2. Token de cliente OAuth2 (client_credentials)

Para comunicações **service-to-service** nas quais não há um usuário humano envolvido,
Primebrick suporta o fluxo **OAuth2 client_credentials**. Este é o mecanismo recomendado para:

- Microsserviços que chamam a API do backend
- Integrações server-side automatizadas
- Jobs e processos batch

### Fluxo

```bash
# Requisição de um token de cliente
curl -X POST https://casdoor.example.com/api/login/oauth/access_token \
  -d "grant_type=client_credentials" \
  -d "client_id=SERVICE_CLIENT_ID" \
  -d "client_secret=SERVICE_CLIENT_SECRET"
```

A resposta contém um JWT que pode ser usado como bearer token:

```bash
curl -H "Authorization: Bearer <jwt>" \
  http://localhost:3001/api/v1/users
```

Os tokens de cliente têm permissões associadas à **service account** correspondente, não a um
usuário humano.

## 3. Chaves API

As chaves API são um mecanismo de autenticação simples para scripts, integrações leves e o
**API Explorer**. Uma chave API é uma string estática associada a um usuário ou service account.

### Uso

```bash
curl -H "X-API-Key: pbk_live_abc123def456..." \
  http://localhost:3001/api/v1/users
```

### Características

- **Não expiram** automaticamente (devem ser revogadas manualmente)
- **Identificam um usuário ou service account** com suas permissões RBAC correspondentes
- **São ideais para scripts** e integrações onde o fluxo OIDC é excessivo

### Gestão das chaves API

As chaves API podem ser criadas e revogadas a partir da interface de administração do Primebrick. É
uma boa prática:

- Usar chaves separadas para cada integração
- Revogar as chaves que não são mais utilizadas
- Não compartilhar chaves entre serviços diferentes

## API Explorer e chaves API

O **API Explorer** (baseado em Scalar) suporta autenticação através de chaves API. Isso permite
testar os endpoints de forma interativa sem ter que gerenciar o fluxo OIDC completo. Configure
sua chave API na interface do Scalar e as requisições "try it" incluirão automaticamente o
header `X-API-Key`.

## Endpoints públicos

Alguns endpoints não requerem autenticação:

- `GET /api/v1/health` — health check
- `GET /api/v1/openapi.json` — especificação OpenAPI do backend
- `GET /api/v1/openapi/aggregated.json` — especificação OpenAPI agregada
