# Microservice-Standard

Primebrick definiert einen Standard für Microservices, der Selbstregistrierung, Heartbeats, Proxy-Routing und OpenAPI-Dokumentation umfasst.

## Überblick

```text
┌──────────────────┐
│   Microservice   │
│  (Users-Service) │
└────────┬─────────┘
         │
         │ 1. Selbstregistrierung (POST /api/v1/services/register)
         ▼
┌──────────────────┐
│   Backend (API)  │
│  Service Registry│
│                  │
│  ┌────────────┐  │
│  │  Registry  │  │  Services + Instanzen
│  └────────────┘  │
│                  │
│  ┌────────────┐  │
│  │  Heartbeat │  │  Lebenszeichen-Überwachung
│  │  Monitor   │  │
│  └────────────┘  │
└────────┬─────────┘
         │
         │ 2. Heartbeats (alle 30s)
         ▼
┌──────────────────┐
│   Backend (API)  │
│  Round-Robin     │
│  Proxy           │
│  /ws/:serviceCode│
└──────────────────┘
         │
         │ 3. Proxy-Requests
         ▼
┌──────────────────┐
│   Microservice   │
│  (Instanz 1/2/3) │
└──────────────────┘
```

## Selbstregistrierung

Jeder Microservice registriert sich beim Start beim Backend. Dies erfolgt über einen POST-Request an den Registrierungs-Endpunkt.

### Registrierungs-Request

```http
POST /api/v1/services/register
Content-Type: application/json
Authorization: Bearer <service-token>

{
  "serviceCode": "users",
  "serviceName": "Users Microservice",
  "version": "1.2.0",
  "instanceId": "users-instance-01",
  "host": "192.168.1.50",
  "port": 3002,
  "healthEndpoint": "/health",
  "openApiEndpoint": "/openapi.json",
  "metadata": {
    "region": "eu-west-1",
    "tier": "standard"
  }
}
```

### Registrierungs-Response

```json
{
  "serviceId": "svc_abc123",
  "instanceId": "users-instance-01",
  "registeredAt": "2025-01-15T10:30:00Z",
  "heartbeatIntervalSeconds": 30
}
```

Nach der Registrierung ist der Microservice über den Backend-Proxy erreichbar.

## Heartbeats

Registrierte Microservices senden regelmäßige **Heartbeats** an das Backend, um ihre Verfügbarkeit zu signalisieren.

### Heartbeat-Request

```http
POST /api/v1/services/heartbeat
Content-Type: application/json
Authorization: Bearer <service-token>

{
  "instanceId": "users-instance-01",
  "status": "healthy",
  "timestamp": "2025-01-15T10:30:30Z",
  "metrics": {
    "cpu": 12.5,
    "memory": 45.2,
    "activeRequests": 3
  }
}
```

### Heartbeat-Intervall

Das Standard-Intervall beträgt **30 Sekunden**. Wird innerhalb von **90 Sekunden** (3 verpasste Heartbeats) kein Heartbeat empfangen, gilt die Instanz als **offline** und wird aus dem Proxy-Rotation entfernt.

```text
Heartbeat verpasst:
  0s   – Letzter Heartbeat empfangen
  30s  – Heartbeat fällig (nicht empfangen)
  60s  – Heartbeat fällig (nicht empfangen)
  90s  – Instanz wird als OFFLINE markiert
```

### Wiederherstellung

Wenn eine offline-Instanz wieder Heartbeats sendet, wird sie automatisch wieder in die Proxy-Rotation aufgenommen.

## Round-Robin-Proxy

Das Backend fungiert als **Reverse Proxy** für Microservices. Requests an `/api/v1/ws/:serviceCode/*` werden mit Round-Robin an die verfügbaren Instanzen weitergeleitet.

### Beispiel

```text
Request:  GET /api/v1/ws/users/users?page=1

Backend:
  1. serviceCode = "users"
  2. Online-Instanzen: [instance-01, instance-02, instance-03]
  3. Round-Robin-Auswahl: instance-02
  4. Weiterleitung: http://192.168.1.51:3002/users?page=1
  5. Response an Client zurückgeben
```

### Lastverteilung

Bei mehreren Instanzen desselben Microservices werden Requests gleichmäßig verteilt:

```text
Instanz 1 → Request 1, Request 4, Request 7, ...
Instanz 2 → Request 2, Request 5, Request 8, ...
Instanz 3 → Request 3, Request 6, Request 9, ...
```

### Fehlertoleranz

Wenn eine Instanz nicht antwortet (Timeout oder Verbindungsfehler), versucht der Proxy die nächste Instanz:

```text
1. Versuch: instance-02 → Timeout (5s)
2. Versuch: instance-03 → Erfolg
3. instance-02 wird als "degraded" markiert
```

## NATS – Message Broker

**NATS** wird für asynchrone Kommunikation zwischen Microservices verwendet. Es unterstützt Publish/Subscribe und Request/Reply-Muster.

### Publish/Subscribe

```typescript
// Microservice A: Event veröffentlichen
import { connect } from 'nats';

const nc = await connect({ servers: 'nats://localhost:4222' });
nc.publish('users.created', JSON.stringify({ userId: '123' }));

// Microservice B: Event abonnieren
const sub = nc.subscribe('users.created');
for await (const msg of sub) {
  const data = JSON.parse(msg.string());
  console.log('Neuer Benutzer erstellt:', data.userId);
}
```

### Request/Reply

```typescript
// Microservice A: Anfrage senden
const response = await nc.request(
  'users.getById',
  JSON.stringify({ userId: '123' }),
  { timeout: 5000 },
);
const user = JSON.parse(response.string());

// Microservice B: Anfrage beantworten
nc.subscribe('users.getById', {
  callback: (err, msg) => {
    const { userId } = JSON.parse(msg.string());
    const user = getUserById(userId);
    msg.respond(JSON.stringify(user));
  },
});
```

## OpenAPI-Dokumentation

Jeder Microservice muss eine **OpenAPI-Spezifikation** bereitstellen. Diese wird vom Backend aggregiert und im API-Explorer angezeigt.

### Anforderungen

1. **OpenAPI-Endpunkt**: Jeder Microservice muss `/openapi.json` (oder einen konfigurierbaren Endpunkt) bereitstellen.
2. **OpenAPI 3.0+**: Die Spec muss OpenAPI 3.0 oder neuer entsprechen.
3. **Pfad-Präfix**: Die Pfade in der Spec sollten ohne das `/ws/:serviceCode`-Präfix sein – das Backend fügt es automatisch hinzu.

### Aggregation

Das Backend aggregiert alle OpenAPI-Specs der online Microservices:

```http
GET /api/v1/openapi/aggregated.json
```

Die aggregierte Spec enthält:

- Alle Backend-Endpunkte
- Alle Microservice-Endpunkte (mit `/ws/:serviceCode`-Präfix)
- Nur Endpunkte von **online** Instanzen

### Beispiel

Microservice `users` mit OpenAPI-Spec:

```json
{
  "openapi": "3.0.3",
  "info": {
    "title": "Users Microservice",
    "version": "1.2.0"
  },
  "paths": {
    "/users": {
      "get": { "summary": "Alle Benutzer abrufen" },
      "post": { "summary": "Benutzer erstellen" }
    },
    "/users/{id}": {
      "get": { "summary": "Benutzer nach ID abrufen" }
    }
  }
}
```

In der aggregierten Spec erscheinen diese Pfade als:

```json
{
  "paths": {
    "/ws/users/users": {
      "get": { "summary": "Alle Benutzer abrufen" },
      "post": { "summary": "Benutzer erstellen" }
    },
    "/ws/users/users/{id}": {
      "get": { "summary": "Benutzer nach ID abrufen" }
    }
  }
}
```
