# RBAC – Role-Based Access Control

Primebrick verwendet ein **Wildcard-Berechtigungssystem** für die feingranulare Zugriffskontrolle. Rollen und Berechtigungen werden in **Casdoor** verwaltet.

## Berechtigungsformat

Berechtigungen folgen dem Format `modul.aktion.granularität`:

```text
modul.aktion.granularität
  │      │        │
  │      │        └── Optionale Granularität (z. B. Feld, Region)
  │      └── Erlaubte Aktion (read, write, create, delete, ...)
  └── Betroffenes Modul (users, billing, invoices, ...)
```

### Beispiele

| Berechtigung | Bedeutung |
|-------------|-----------|
| `users.read` | Lesezugriff auf das Users-Modul |
| `users.write` | Schreibzugriff auf das Users-Modul |
| `users.create` | Benutzer erstellen |
| `users.delete` | Benutzer löschen |
| `billing.read` | Lesezugriff auf das Billing-Modul |
| `invoices.read.paid` | Lesezugriff auf bezahlte Rechnungen |
| `invoices.write` | Schreibzugriff auf Rechnungen |

## Wildcards

Wildcards (`*`) können auf jeder Ebene verwendet werden:

| Berechtigung | Bedeutung |
|-------------|-----------|
| `users.*` | Alle Aktionen im Users-Modul |
| `*.read` | Lesezugriff auf alle Module |
| `*` | Vollzugriff auf alles |

### Beispiele für Wildcard-Matching

```text
Berechtigung: users.*
  ✓ users.read     – Match
  ✓ users.write    – Match
  ✓ users.delete   – Match
  ✗ billing.read   – Kein Match (anderes Modul)

Berechtigung: *.read
  ✓ users.read     – Match
  ✓ billing.read   – Match
  ✗ users.write    – Kein Match (andere Aktion)

Berechtigung: *
  ✓ users.read     – Match
  ✓ billing.write  – Match
  ✓ anything.any   – Match
```

## Rollenzuordnungen mit Casdoor

Rollen werden in **Casdoor** definiert und Benutzern zugewiesen. Jede Rolle enthält eine Menge von Berechtigungen.

### Rollen-Definition

In Casdoor wird eine Rolle mit folgenden Eigenschaften definiert:

```text
Rolle: "user-manager"
Berechtigungen:
  - users.read
  - users.write
  - users.create
  - users.delete
```

### Benutzer-Rollen-Zuweisung

Ein Benutzer kann mehrere Rollen haben. Die effektiven Berechtigungen sind die **Vereinigung** aller Berechtigungen aus allen Rollen:

```text
Benutzer: "alice@example.com"
Rollen:
  - user-manager  → users.read, users.write, users.create, users.delete
  - billing-viewer → billing.read, invoices.read
Effektive Berechtigungen:
  - users.read, users.write, users.create, users.delete
  - billing.read, invoices.read
```

## Berechtigungsprüfung im Backend

Bei jedem Request prüft das Backend die Berechtigungen des Benutzers gegen die erforderliche Berechtigung des Endpunkts.

### Ablauf

```text
1. Request trifft ein:  GET /api/v1/ws/users/users
2. Auth Guard:          JWT validieren → Benutzer identifizieren
3. RBAC Guard:          Erforderliche Berechtigung: "users.read"
4. Prüfung:             Hat der Benutzer "users.read" oder "users.*" oder "*.read" oder "*"?
5. Ergebnis:            ✓ Erlaubt → Request wird verarbeitet
                        ✗ Verweigert → 403 Forbidden
```

### Beispiel-Code (konzeptionell)

```typescript
function checkPermission(
  userPermissions: string[],
  requiredPermission: string,
): boolean {
  const [reqModule, reqAction, reqGranularity] = requiredPermission.split('.');

  for (const perm of userPermissions) {
    const [mod, action, granularity] = perm.split('.');

    const moduleMatch = mod === '*' || mod === reqModule;
    const actionMatch = action === '*' || action === reqAction;
    const granMatch =
      !reqGranularity ||
      granularity === '*' ||
      granularity === reqGranularity;

    if (moduleMatch && actionMatch && granMatch) {
      return true;
    }
  }

  return false;
}
```

## Admin-Bypass

Benutzer mit der Rolle **`admin`** erhalten automatisch Vollzugriff auf alle Endpunkte, ohne dass spezifische Berechtigungen geprüft werden.

```text
Benutzer: "admin@example.com"
Rollen: ["admin"]
→ Admin-Bypass aktiviert
→ Alle Berechtigungsprüfungen werden übersprungen
```

Dies vereinfacht die Verwaltung, da Admin-Benutzer nicht jede einzelne Berechtigung zugewiesen werden muss.

### Admin-Rolle in Casdoor

Die `admin`-Rolle ist eine reservierte Rolle. Wenn sie einem Benutzer zugewiesen wird, aktiviert sie den Admin-Bypass im Backend.

## Fehler-Responses

Bei fehlenden Berechtigungen gibt das Backend einen **403 Forbidden**-Response zurück:

```json
{
  "type": "https://primebrick.dev/errors/forbidden",
  "title": "Zugriff verweigert",
  "status": 403,
  "detail": "Sie haben nicht die erforderliche Berechtigung 'users.read', um auf diese Ressource zuzugreifen.",
  "code": "RBAC_PERMISSION_DENIED",
  "required_permission": "users.read"
}
```

Siehe [Fehlerbehandlung](/de/api/error-handling/) für Details zum Fehlerformat.
