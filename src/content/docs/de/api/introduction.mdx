# API-Einführung

Die Primebrick REST-API ist die zentrale Schnittstelle für die Kommunikation mit der Plattform. Alle Endpunkte sind unter dem Präfix `/api/v1/` erreichbar.

## Basis-URLs

Je nach Umgebung ändert sich die Basis-URL:

| Umgebung | Basis-URL |
|----------|-----------|
| Lokale Entwicklung | `http://localhost:3001` |
| Staging | `https://staging-api.primebrick.dev` |
| Produktion | `https://api.primebrick.dev` |

Alle API-Endpunkte sind unter `/api/v1/` erreichbar. Beispiel:

```text
GET https://api.primebrick.dev/api/v1/users
```

## Endpunkt-Struktur

### Backend-Endpunkte

Endpunkte, die direkt vom Backend bereitgestellt werden:

```text
GET    /api/v1/health                    – Health-Check
GET    /api/v1/openapi.json              – OpenAPI-Spec (Backend)
GET    /api/v1/openapi/aggregated.json   – Aggregierte OpenAPI-Spec
GET    /api/v1/services                  – Registrierte Microservices
```

### Microservice-Endpunkte

Endpunkte von Microservices sind unter `/ws/:serviceCode` erreichbar. Der `serviceCode` entspricht dem bei der Registrierung angegebenen Code.

```text
GET    /api/v1/ws/users/users            – Users-Microservice: Liste aller Benutzer
POST   /api/v1/ws/users/users            – Users-Microservice: Benutzer erstellen
GET    /api/v1/ws/billing/invoices       – Billing-Microservice: Rechnungen
```

Der Backend-Proxy leitet Requests mit Round-Robin an die registrierten Instanzen des jeweiligen Microservices weiter.

## Versionierung

Die API verwendet **URI-Versionierung** mit dem Präfix `/api/v1/`. Bei breaking changes wird eine neue Version (`/api/v2/`) eingeführt, während die alte Version weiterhin verfügbar bleibt.

## OpenAPI-Spezifikation

Primebrick stellt zwei OpenAPI-Spezifikationen bereit:

1. **Backend-Spec** (`/api/v1/openapi.json`): Enthält nur die Backend-Endpunkte.
2. **Aggregierte Spec** (`/api/v1/openapi/aggregated.json`): Enthält die Backend-Endpunkte sowie alle Endpunkte der online Microservices, mit dem Präfix `/ws/:serviceCode`.

Die aggregierte Spec ist öffentlich zugänglich und erfordert keine Authentifizierung.

## API-Explorer

Der interaktive [API-Explorer](/api-explorer/) verwendet die aggregierte OpenAPI-Spec und ermöglicht das Testen von Endpunkten direkt im Browser. Die Server-URL und Authentifizierung sind vom Benutzer konfigurierbar.

## Content-Type

Alle Requests und Responses verwenden **JSON**:

```http
Content-Type: application/json
Accept: application/json
```
