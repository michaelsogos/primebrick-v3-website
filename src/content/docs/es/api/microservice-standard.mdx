## Visión general

Todo microservicio de Primebrick sigue un **contrato estándar** aplicado por `@primebrick/sdk`. Esto garantiza un comportamiento uniforme en todos los servicios — independientemente de la lógica de negocio que implementen — y permite al backend gestionarlos de forma genérica.

## El contrato estándar

Un microservicio conforme debe:

1. **Autoregistrarse** en el backend al iniciarse
2. **Enviar heartbeats** a intervalos regulares
3. **Exponer OpenAPI** en `/api/openapi.json`
4. **Usar NATS** para la mensajería entre servicios
5. **Devolver RFC 7807 Problem Details** para todos los errores

## Autoregistro

Al iniciarse, un microservicio se registra en el backend usando el SDK:

```typescript
import { Microservice } from '@primebrick/sdk';

const service = new Microservice({
  serviceCode: 'billing',
  baseUrl: 'http://localhost:3002',
  healthEndpoint: '/health',
  openapiEndpoint: '/api/openapi.json',
  natsUrl: 'nats://localhost:4222',
});

await service.start();
```

El payload de registro incluye:

| Campo | Descripción |
|-------|-------------|
| `serviceCode` | Identificador único del servicio (p. ej. `billing`) |
| `baseUrl` | URL donde la instancia es accesible |
| `healthEndpoint` | Ruta del health check |
| `openapiEndpoint` | Ruta de la especificación OpenAPI |

## Heartbeats y detección de instancias obsoletas

Cada instancia registrada envía un **heartbeat** al backend a un intervalo configurable (por defecto: cada 15 segundos).

El backend registra el último heartbeat de cada instancia. Si una instancia no envía un heartbeat dentro del **umbral de obsolescencia** (por defecto: 60 segundos), se:

1. Marca como **obsoleta** (stale) en el service registry
2. Elimina de la rotación del **proxy round-robin**
3. Excluye de la **especificación OpenAPI agregada**

Esto garantiza que las instancias caídas o inalcanzables se detecten automáticamente sin intervención manual.

## Proxy round-robin

El backend expone un proxy transparente en **`/ws/:serviceCode`** que reenvía las peticiones a las instancias registradas del microservicio solicitado.

```text
GET  /ws/billing/invoices        → proxied al microservicio billing
POST /ws/inventory/items         → proxied al microservicio inventory
```

Cuando hay múltiples instancias de un servicio online, el proxy distribuye las peticiones usando balanceo **round-robin**. Los clientes no necesitan saber cuántas instancias existen ni dónde están — siempre se conectan al backend y el proxy gestiona el enrutamiento.

## Mensajería NATS

Los microservicios se comunican entre sí y con el backend a través de **NATS**. El SDK proporciona helpers para patrones comunes:

```typescript
// Publicar un evento
await service.nats.publish('billing.invoice.created', { invoiceId: 123 });

// Suscribirse a eventos
service.nats.subscribe('inventory.item.updated', async (msg) => {
  await updatePricing(msg.data.itemId);
});

// Request/reply (estilo síncrono sobre NATS)
const result = await service.nats.request('inventory.check-stock', {
  itemId: 'SKU-001',
  quantity: 10,
});
```

NATS desacopla los servicios: un servicio puede publicar eventos sin saber quién los consume, y múltiples instancias pueden suscribirse al mismo subject para escalado horizontal.

## Exposición OpenAPI

Cada microservicio debe exponer su especificación OpenAPI en:

```text
GET /api/openapi.json
```

El backend obtiene la especificación de cada servicio online y la combina en la **especificación agregada** en `/api/v1/openapi/aggregated.json`. Las rutas de los microservicios se prefijan con `/ws/:serviceCode` para que coincidan con el proxy.

Si un servicio está offline, su especificación simplemente se omite de la agregación.

## Manejo de errores RFC 7807

Todos los errores — del backend y de los microservicios — usan **RFC 7807 Problem Details**. El SDK proporciona un helper para producir errores conformes:

```typescript
import { ProblemError } from '@primebrick/sdk';

throw new ProblemError({
  type: 'https://primebrick.dev/errors/invoice-not-found',
  title: 'Invoice not found',
  status: 404,
  detail: `Invoice ${invoiceId} does not exist.`,
  instance: `/ws/billing/invoices/${invoiceId}`,
});
```

Ver [Manejo de errores](./error-handling/) para el formato completo.

## Lista de comprobación

Un microservicio es conforme cuando:

- [x] Se registra vía `@primebrick/sdk` al iniciarse
- [x] Envía heartbeats y gestiona la detección de obsolescencia
- [x] Es accesible a través del proxy `/ws/:serviceCode`
- [x] Usa NATS para la mensajería entre servicios
- [x] Expone OpenAPI en `/api/openapi.json`
- [x] Devuelve RFC 7807 Problem Details para todos los errores
