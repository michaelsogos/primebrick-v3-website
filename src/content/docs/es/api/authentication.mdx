# Autenticación

Primebrick utiliza **Casdoor** como Identity Provider (IdP) externo. La autenticación se basa
en el protocolo **OIDC** (OpenID Connect) y los tokens emitidos son **JWT** (JSON Web Token).

## Visión general

Existen **tres tipos de credenciales** para autenticarse en la API de Primebrick:

| Tipo | Caso de uso | Header |
|------|-------------|--------|
| **Token de usuario** | Aplicaciones frontend (usuarios reales) | `Authorization: Bearer <jwt>` |
| **Token de cliente OAuth2** | Comunicaciones service-to-service | `Authorization: Bearer <jwt>` |
| **Clave API** | Scripts, integraciones, API Explorer | `X-API-Key: <api-key>` |

Los tres mecanismos son validados por el backend, que verifica la identidad y los permisos
asociados a la credencial antes de procesar la petición.

## 1. Token de usuario (JWT)

El flujo primario de autenticación para las aplicaciones frontend es el **flujo OIDC
Authorization Code**. El usuario es redirigido a Casdoor para iniciar sesión, y recibe a cambio
un JWT que se envía en cada petición a la API.

### Flujo

1. El frontend redirige al usuario a la página de login de Casdoor
2. El usuario introduce sus credenciales en Casdoor
3. Casdoor redirige al frontend con un authorization code
4. El frontend intercambia el authorization code por un JWT (access token)
5. El frontend envía el JWT en cada petición a la API

### Uso

```bash
curl -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..." \
  http://localhost:3001/api/v1/users
```

El backend valida el JWT verificando:

- La **firma** utilizando las claves públicas de Casdoor (JWKS)
- La **expiración** (claim `exp`)
- El **emisor** (claim `iss`) corresponde a Casdoor
- La **audiencia** (claim `aud`) corresponde a la aplicación Primebrick

### Refresh token

Los JWT tienen una expiración limitada (típicamente 1 hora). El frontend utiliza el refresh
token para obtener un nuevo JWT sin requerir que el usuario inicie sesión de nuevo.

## 2. Token de cliente OAuth2 (client_credentials)

Para las comunicaciones **service-to-service** en las que no hay un usuario humano involucrado,
Primebrick soporta el flujo **OAuth2 client_credentials**. Este es el mecanismo recomendado para:

- Microservicios que llaman a la API del backend
- Integraciones server-side automatizadas
- Jobs y procesos batch

### Flujo

```bash
# Petición de un token de cliente
curl -X POST https://casdoor.example.com/api/login/oauth/access_token \
  -d "grant_type=client_credentials" \
  -d "client_id=SERVICE_CLIENT_ID" \
  -d "client_secret=SERVICE_CLIENT_SECRET"
```

La respuesta contiene un JWT que puede usarse como bearer token:

```bash
curl -H "Authorization: Bearer <jwt>" \
  http://localhost:3001/api/v1/users
```

Los tokens de cliente tienen permisos asociados a la **service account** correspondiente, no a un
usuario humano.

## 3. Claves API

Las claves API son un mecanismo de autenticación sencillo para scripts, integraciones ligeras y
el **API Explorer**. Una clave API es una cadena estática asociada a un usuario o service account.

### Uso

```bash
curl -H "X-API-Key: pbk_live_abc123def456..." \
  http://localhost:3001/api/v1/users
```

### Características

- **No expiran** automáticamente (deben revocarse manualmente)
- **Identifican a un usuario o service account** con sus permisos RBAC correspondientes
- **Son ideales para scripts** e integraciones donde el flujo OIDC es excesivo

### Gestión de las claves API

Las claves API pueden crearse y revocarse desde la interfaz de administración de Primebrick. Es
una buena práctica:

- Usar claves separadas para cada integración
- Revocar las claves que ya no se utilicen
- No compartir claves entre servicios diferentes

## API Explorer y claves API

El **API Explorer** (basado en Scalar) soporta la autenticación mediante claves API. Esto permite
probar los endpoints de forma interactiva sin tener que gestionar el flujo OIDC completo. Configura
tu clave API en la interfaz de Scalar y las peticiones "try it" incluirán automáticamente el
header `X-API-Key`.

## Endpoints públicos

Algunos endpoints no requieren autenticación:

- `GET /api/v1/health` — health check
- `GET /api/v1/openapi.json` — especificación OpenAPI del backend
- `GET /api/v1/openapi/aggregated.json` — especificación OpenAPI agregada
