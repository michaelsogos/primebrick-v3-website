# Gestione errori

L'API di Primebrick adotta lo standard **RFC 7807** (Problem Details for HTTP APIs) per
tutte le risposte di errore. Questo garantisce un formato coerente, prevedibile e
facilmente gestibile dai client.

## RFC 7807 Problem Details

Ogni risposta di errore ha il content-type `application/problem+json` e segue la struttura
definita da RFC 7807:

```json
{
  "type": "https://api.primebrick.dev/errors/permission-denied",
  "title": "Permission Denied",
  "status": 403,
  "detail": "Non hai il permesso necessario per completare questa operazione.",
  "instance": "/api/v1/users/usr_abc123",
  "errorCode": "PERMISSION_DENIED",
  "impact": "La risorsa richiesta non può essere modificata."
}
```

### Campi standard

| Campo | Tipo | Descrizione |
|-------|------|-------------|
| `type` | string (URI) | URI che identifica il tipo di problema |
| `title` | string | Breve descrizione leggibile del tipo di errore |
| `status` | number | Codice di stato HTTP |
| `detail` | string | Descrizione specifica di questa occorrenza dell'errore |
| `instance` | string (URI) | URI che identifica la specifica occorrenza (tipicamente il path) |

### Campi estesi Primebrick

Primebrick estende lo standard con due campi aggiuntivi:

| Campo | Tipo | Descrizione |
|-------|------|-------------|
| `errorCode` | string | Codice di errore stabile e indipendente dalla localizzazione |
| `impact` | string | Descrizione dell'impatto dell'errore sull'operazione richiesta |

## Codici di errore stabili

Il campo **`errorCode`** è un identificatore stabile, in maiuscolo e con underscore, che
i client possono utilizzare per gestire gli errori in modo programmatico. A differenza di
`title` e `detail` (che possono variare in base alla lingua), `errorCode` è **garantito
stabile** tra le versioni.

### Codici di errore comuni

| `errorCode` | HTTP | Descrizione |
|-------------|------|-------------|
| `UNAUTHORIZED` | 401 | Credenziali mancanti o non valide |
| `PERMISSION_DENIED` | 403 | Permesso RBAC mancante |
| `NOT_FOUND` | 404 | Risorsa non trovata |
| `VALIDATION_ERROR` | 422 | Dati della richiesta non validi |
| `CONFLICT` | 409 | Conflitto (es. risorsa duplicata) |
| `RATE_LIMITED` | 429 | Limite di richieste superato |
| `SERVICE_UNAVAILABLE` | 503 | Microservizio non raggiungibile o offline |
| `INTERNAL_ERROR` | 500 | Errore interno del server |

### Esempio: errore di validazione

```json
{
  "type": "https://api.primebrick.dev/errors/validation-error",
  "title": "Validation Error",
  "status": 422,
  "detail": "Il campo 'email' non è un indirizzo email valido.",
  "instance": "/api/v1/users",
  "errorCode": "VALIDATION_ERROR",
  "impact": "L'utente non è stato creato. Correggi i dati e riprova.",
  "errors": [
    {
      "field": "email",
      "message": "Formato email non valido"
    }
  ]
}
```

Per gli errori di validazione, il campo aggiuntivo `errors` contiene l'elenco dettagliato
dei campi non validi.

### Esempio: microservizio non disponibile

```json
{
  "type": "https://api.primebrick.dev/errors/service-unavailable",
  "title": "Service Unavailable",
  "status": 503,
  "detail": "Il microservizio 'document-service' non è attualmente disponibile.",
  "instance": "/api/v1/ws/document-service/documents",
  "errorCode": "SERVICE_UNAVAILABLE",
  "impact": "Le operazioni sui documenti non sono temporaneamente disponibili. Riprova più tardi."
}
```

## Il campo `impact`

Il campo **`impact`** è un'estensione di Primebrick che descrive **l'impatto concreto**
dell'errore sull'operazione richiesta. Aiuta gli sviluppatori e gli utenti finali a
comprendere:

- **Cosa non è successo** a causa dell'errore
- **Quali azioni** possono intraprendere per risolvere o aggirare il problema

### Esempi di `impact`

| Errore | `impact` |
|--------|----------|
| `PERMISSION_DENIED` | "La risorsa richiesta non può essere modificata." |
| `SERVICE_UNAVAILABLE` | "Le operazioni sui documenti non sono temporaneamente disponibili. Riprova più tardi." |
| `VALIDATION_ERROR` | "L'utente non è stato creato. Correggi i dati e riprova." |
| `CONFLICT` | "Esiste già un utente con questa email. Utilizza un'email diversa." |

## Codici di stato HTTP

Primebrick utilizza i codici di stato HTTP standard in modo coerente:

| Codice | Significato | Quando |
|--------|-------------|--------|
| `200` | OK | Richiesta completata con successo |
| `201` | Created | Risorsa creata con successo |
| `204` | No Content | Operazione completata, nessun contenuto da restituire |
| `400` | Bad Request | Sintassi della richiesta non valida |
| `401` | Unauthorized | Autenticazione mancante o non valida |
| `403` | Forbidden | Autenticato ma senza permessi sufficienti |
| `404` | Not Found | Risorsa non trovata |
| `409` | Conflict | Conflitto con lo stato attuale (es. duplicato) |
| `422` | Unprocessable Entity | Validazione fallita |
| `429` | Too Many Requests | Rate limiting |
| `500` | Internal Server Error | Errore generico lato server |
| `503` | Service Unavailable | Microservizio non raggiungibile |

## Gestione degli errori nel client

I client dovrebbero sempre gestire le risposte di errore verificando il `errorCode` per
una logica specifica, e mostrando `detail` o `impact` all'utente:

```typescript
try {
  const response = await api.users.create({ email: 'invalid' });
} catch (error) {
  if (error.errorCode === 'VALIDATION_ERROR') {
    // Mostra gli errori di validazione campo per campo
    showValidationErrors(error.errors);
  } else if (error.errorCode === 'CONFLICT') {
    // Gestione specifica del conflitto
    showToast(error.impact);
  } else {
    // Errore generico
    showToast(error.detail);
  }
}
```
