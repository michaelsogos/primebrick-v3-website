# RBAC — Role-Based Access Control

Primebrick implementa un modello di controllo degli accessi basato sui ruoli (**RBAC**) con
permessi strutturati secondo una notazione **wildcard**. L'autorizzazione è gestita in
collaborazione con **Casdoor**, che mantiene i role mappings tra utenti, ruoli e permessi.

## Permessi wildcard

I permessi in Primebrick seguono il formato **`modulo.azione.granularità`**:

```text
modulo.azione.granularità
  │       │        │
  │       │        └── livello di granularità (opzionale, può essere *)
  │       └── azione specifica (read, write, delete, ...)
  └── modulo o risorsa (users, documents, settings, ...)
```

### Esempi

| Permesso | Significato |
|----------|-------------|
| `users.read` | Può leggere gli utenti |
| `users.read.*` | Può leggere gli utenti a qualsiasi livello di granularità |
| `users.write` | Può modificare gli utenti |
| `users.delete` | Può eliminare gli utenti |
| `documents.read.own` | Può leggere solo i propri documenti |
| `documents.read.department` | Può leggere i documenti del proprio dipartimento |
| `documents.read.*` | Può leggere tutti i documenti |
| `*.*.*` | Accesso completo (admin bypass) |

### Wildcard

Il carattere `*` funge da wildcard e può apparire in qualsiasi posizione:

- `users.*` — tutte le azioni sul modulo `users`
- `*.read` — azione `read` su tutti i moduli
- `*.*.*` o `*` — accesso completo a tutto

### Granularità

Il terzo livello (`granularità`) definisce **l'ambito** del permesso. I valori comuni sono:

| Granularità | Descrizione |
|-------------|-------------|
| `own` | Solo le risorse create dall'utente |
| `department` | Le risorse del dipartimento dell'utente |
| `organization` | Le risorse dell'organizzazione dell'utente |
| `*` | Tutte le risorse, senza restrizioni |

Se la granularità è omessa, si intende `*` (nessuna restrizione).

## Role mappings con Casdoor

**Casdoor** gestisce la mappatura tra utenti, ruoli e permessi. Il flusso è il seguente:

1. Un **utente** viene assegnato a uno o più **ruoli** in Casdoor
2. Ogni **ruolo** ha un set di **permessi** associati
3. Quando l'utente si autentica, il JWT contiene i ruoli assegnati
4. Il backend estrae i permessi dai ruoli e verifica l'accesso

### Struttura in Casdoor

```text
Utente: m.rossi@example.com
  └── Ruoli: [admin, editor]
        ├── admin → [*.*.*]
        └── editor → [documents.read.*, documents.write.own, users.read]
```

### Token JWT

Il JWT emesso da Casdoor include i ruoli dell'utente nel claim appropriato. Il backend
risolve i ruoli nei permessi corrispondenti e li memorizza nel contesto della richiesta
per i controlli di autorizzazione.

## Verifica dei permessi nel backend

Il backend verifica i permessi ad ogni richiesta. La logica di verifica confronta il
permesso richiesto dall'endpoint con i permessi effettivi dell'utente:

```typescript
// Esempio concettuale della verifica
function hasPermission(userPermissions: string[], required: string): boolean {
  return userPermissions.some(p => matchWildcard(p, required));
}

// Verifica se "users.read.*" copre "users.read.own"
hasPermission(['users.read.*'], 'users.read.own'); // true

// Verifica se "*.read" copre "documents.read"
hasPermission(['*.read'], 'documents.read'); // true
```

Il matching delle wildcard segue regole standard: `*` in una posizione corrisponde a
qualsiasi valore in quella posizione.

## Admin bypass

Gli utenti con il ruolo **admin** (o con il permesso `*.*.*`) hanno **accesso completo**
a tutte le risorse e tutte le azioni, bypassando tutti i controlli di autorizzazione.

Questo è implementato come prima verifica nel middleware di autorizzazione:

```text
SE l'utente ha il permesso *.*.*:
  → consenti l'accesso (bypass completo)
ALTRIMENTI:
  → verifica il permesso specifico richiesto dall'endpoint
```

## Endpoint e permessi richiesti

Ogni endpoint del backend dichiara il permesso richiesto. Ad esempio:

| Endpoint | Permesso richiesto |
|----------|-------------------|
| `GET /api/v1/users` | `users.read` |
| `POST /api/v1/users` | `users.write` |
| `DELETE /api/v1/users/:id` | `users.delete` |
| `GET /api/v1/ws/document-service/documents` | `documents.read` |

Se l'utente non ha il permesso richiesto, il backend restituisce un errore **403 Forbidden**
con un Problem Details che indica il permesso mancante (vedi [Gestione errori](/it/api/error-handling/)).
