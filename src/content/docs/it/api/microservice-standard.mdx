# Standard microservizi

Primebrick definisce uno **standard rigoroso** per i microservizi. Ogni microservizio che
vuole integrarsi nell'ecosistema Primebrick deve seguire queste convenzioni per garantire
scopribilità, bilanciamento del carico e documentazione automatica.

## Panoramica

Ogni microservizio Primebrick:

1. Si **autoregistra** presso il backend all'avvio
2. Invia un **heartbeat** periodico per segnalare che è attivo
3. Espone un'**API REST** documentata con **OpenAPI**
4. Viene **proxato** dal backend tramite `/ws/:serviceCode/`
5. Può comunicare asincronamente tramite **NATS**

## Autoregistrazione

All'avvio, ogni microservizio contatta l'endpoint di registrazione del backend per
annunciare la propria presenza:

```text
POST /api/v1/service-registry/register
```

Il payload di registrazione include:

```json
{
  "serviceCode": "document-service",
  "serviceName": "Document Service",
  "version": "1.2.0",
  "baseUrl": "http://10.0.0.15:3002",
  "openapiUrl": "http://10.0.0.15:3002/openapi.json",
  "healthUrl": "http://10.0.0.15:3002/health",
  "metadata": {
    "instanceId": "doc-svc-instance-001",
    "region": "eu-west-1"
  }
}
```

Il backend registra il microservizio nel **service registry**, un registro in memoria
(oppure su database) che mantiene l'elenco di tutte le istanze attive per ogni servizio.

### Deregistrazione

Quando un microservizio si arresta in modo pulito, invia una richiesta di deregistrazione:

```text
POST /api/v1/service-registry/deregister
```

Se un microservizio si arresta anomalamante, il backend rileva l'assenza di heartbeat e
rimuove automaticamente l'istanza dal registry (vedi sotto).

## Heartbeat

Ogni microservizio invia un **heartbeat** periodico al backend (tipicamente ogni 10-30
secondi) per confermare che è ancora attivo:

```text
POST /api/v1/service-registry/heartbeat
```

```json
{
  "serviceCode": "document-service",
  "instanceId": "doc-svc-instance-001",
  "timestamp": "2025-01-15T10:30:00Z"
}
```

Il backend utilizza l'heartbeat per:

- **Aggiornare il timestamp** di ultima attività dell'istanza
- **Rimuovere istanze morte** — se un'istanza non invia heartbeat entro un intervallo
  configurato (es. 90 secondi), viene rimossa dal registry
- **Bilanciare il carico** — solo le istanze con heartbeat recente ricevono richieste

## Proxy round-robin

Quando il backend riceve una richiesta indirizzata a un microservizio (sotto
`/ws/:serviceCode/`), cerca nel service registry tutte le istanze attive per quel
`serviceCode` e instrada la richiesta con bilanciamento **round-robin**:

```text
Client → Backend /api/v1/ws/document-service/documents
                    │
                    ├── Istanza 1: http://10.0.0.15:3002  ← richiesta 1
                    ├── Istanza 2: http://10.0.0.16:3002  ← richiesta 2
                    └── Istanza 3: http://10.0.0.17:3002  ← richiesta 3
                                                    richiesta 4 → Istanza 1 (ciclo)
```

Il proxy:

- **Preserva gli header** di autenticazione e autorizzazione
- **Propaga** l'identità dell'utente (dal JWT) al microservizio
- **Gestisce i timeout** — se un'istanza non risponde entro il timeout, restituisce un errore
- **Riprova** su un'altra istanza se quella selezionata non risponde (failover)

### Prefisso dei percorsi

I percorsi OpenAPI dei microservizi vengono prefissati con **`/ws/:serviceCode`** nello
specifica aggregata. Ad esempio, se un microservizio espone `GET /documents`, nell'API
aggregata diventa `GET /api/v1/ws/document-service/documents`.

## NATS

**NATS** è il message broker utilizzato per la comunicazione **asincrona** tra il backend
e i microservizi (e tra microservizi stessi). A differenza del proxy HTTP (sincrono),
NATS è ideale per:

- **Eventi** — notificare altri servizi quando qualcosa accade (es. "utente creato",
  "documento approvato")
- **Job** — accodare elaborazioni asincrone (es. generazione report, export bulk)
- **Comunicazione fire-and-forget** — inviare messaggi senza attendere risposta

### Esempio di pubblicazione evento

```typescript
// Il backend pubblica un evento quando un utente viene creato
nats.publish('user.created', { userId: 'usr_123', email: 'm.rossi@example.com' });
```

### Esempio di sottoscrizione

```typescript
// Un microservizio si sottoscrive all'evento
nats.subscribe('user.created', (data) => {
  // Invia email di benvenuto, crea record aggiuntivi, ecc.
  sendWelcomeEmail(data.email);
});
```

## OpenAPI

Ogni microservizio **deve** esporre un endpoint OpenAPI:

```text
GET /openapi.json
```

Il backend raccoglie le specifiche OpenAPI di tutti i microservizi online e le unisce
con la propria in un'**API aggregata**:

```text
GET /api/v1/openapi/aggregated.json
```

L'API aggregata:

- Include gli endpoint del backend (sotto `/api/v1/`)
- Include gli endpoint di tutti i microservizi online (prefissati con `/ws/:serviceCode`)
- **Esclude** i microservizi offline — se un servizio non è nel registry, i suoi endpoint
  non compaiono
- È **pubblica** — non richiede autenticazione, per consentire all'API Explorer di funzionare

Questo significa che l'API Explorer riflette sempre lo stato attuale del sistema: se un
microservizio è offline, i suoi endpoint scompaiono automaticamente dalla documentazione
interattiva.

## Ciclo di vita di un microservizio

```text
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│   Avvio     │────▶│ Registrazione│────▶│   Heartbeat     │
│             │     │  (POST)      │     │   periodico     │
└─────────────┘     └──────────────┘     └────────┬────────┘
                                                   │
                                           ┌───────▼────────┐
                                           │  Servizio      │
                                           │  operativo     │
                                           │  (proxy + NATS)│
                                           └───────┬────────┘
                                                   │
                                    ┌──────────────┼──────────────┐
                                    ▼                             ▼
                            ┌──────────────┐            ┌──────────────┐
                            │ Deregistrazione│           │  Timeout     │
                            │ (arresto)     │            │  heartbeat   │
                            │  (POST)       │            │  → rimozione │
                            └──────────────┘            └──────────────┘
```
