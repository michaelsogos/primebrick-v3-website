# Architecture

Primebrick adopte une architecture à composants modulaires, conçue pour être évolutive,
portable et sans vendor lock-in. Chaque composant a des responsabilités bien définies et
communique avec les autres via des protocoles standards.

## Backend (Express API)

Le backend est le cœur du système. Il est construit avec **Express** et fournit :

- **API REST** sous le préfixe `/api/v1/`
- **Authentification OIDC** via Casdoor (JWT bearer token)
- **Autorisation RBAC** avec permissions wildcard (`module.action.granularité`)
- **Proxy pour microservices** avec équilibrage round-robin
- **Service registry** pour l'auto-enregistrement des microservices
- **Endpoint OpenAPI agrégé** qui combine les spécifications du backend et de tous les
  microservices en ligne

Le backend est le seul composant exposé directement au frontend. Les microservices ne sont
jamais accessibles directement depuis le client.

## Frontend (SvelteKit)

Le frontend est une application **SvelteKit** qui fournit :

- **UI de backoffice** préconfigurée avec login, dashboards et gestion des données
- **Client typé** généré par le SDK à partir de l'OpenAPI agrégé
- **Gestion de l'authentification** intégrée avec Casdoor (flux OIDC authorization code)
- **Routing** basé sur le système de fichiers de SvelteKit

Le frontend communique exclusivement avec le backend via l'API REST. Il n'accède pas
directement à la base de données, aux microservices ou à Casdoor.

## Microservices

Les microservices sont des services autonomes qui implémentent une logique métier spécifique (ex.
génération de documents, traitement batch, intégrations externes). Chaque microservice :

- **S'auto-enregistre** auprès du backend au démarrage
- Envoie un **heartbeat** périodique pour signaler qu'il est actif
- Expose une **API REST** documentée avec OpenAPI
- Est **proxied** par le backend via le préfixe `/ws/:serviceCode/`

Le backend maintient un **service registry** avec la liste des microservices en ligne et route les
requêtes avec équilibrage **round-robin** lorsqu'il y a plusieurs instances du même service.

La communication asynchrone entre composants se fait via **NATS**, un message broker
léger et haute performance.

## DAL (Data Access Layer)

Le DAL est une **bibliothèque partagée** qui abstrait l'accès aux données. Il fournit :

- **ORM typé** avec support pour PostgreSQL (et autres bases de données relationnelles)
- **Migrations** versionnées et reproductibles
- **Repository pattern** pour isoler la logique de persistance
- **Transactions** déclaratives

Le DAL est utilisé à la fois par le backend et par les microservices, garantissant un accès aux
données cohérent et typé dans tout le système.

## SDK

Le SDK est **généré automatiquement** à partir de l'endpoint OpenAPI agrégé du backend.
Il fournit :

- **Client TypeScript** typé pour toutes les APIs du backend et des microservices
- **Validation** des types à la compilation
- **Régénération** simple lorsque l'API change

Cela élimine le besoin d'écrire des clients API à la main et garantit que le frontend
est toujours aligné avec le backend.

## Flux d'une requête typique

1. L'utilisateur interagit avec le **Frontend** (SvelteKit)
2. Le Frontend envoie une requête HTTP au **Backend** via le SDK typé
3. Le Backend valide le **JWT** avec Casdoor et vérifie les **permissions RBAC**
4. Si la requête est pour un microservice, le Backend la **proxie** vers le service approprié
5. Le microservice traite la requête, accédant à la base de données via le **DAL**
6. La réponse remonte la chaîne jusqu'au Frontend
