# RBAC — Role-Based Access Control

Primebrick implémente un modèle de contrôle d'accès basé sur les rôles (**RBAC**) avec des permissions
structurées selon une notation **wildcard**. L'autorisation est gérée en collaboration avec
**Casdoor**, qui maintient l'attribution des rôles entre utilisateurs, rôles et permissions.

## Permissions wildcard

Les permissions dans Primebrick suivent le format **`module.action.granularité`** :

```text
module.action.granularité
  │       │        │
  │       │        └── niveau de granularité (optionnel, peut être *)
  │       └── action spécifique (read, write, delete, ...)
  └── module ou ressource (users, documents, settings, ...)
```

### Exemples

| Permission | Signification |
|------------|---------------|
| `users.read` | Peut lire les utilisateurs |
| `users.read.*` | Peut lire les utilisateurs à tout niveau de granularité |
| `users.write` | Peut modifier les utilisateurs |
| `users.delete` | Peut supprimer les utilisateurs |
| `documents.read.own` | Peut lire uniquement ses propres documents |
| `documents.read.department` | Peut lire les documents de son département |
| `documents.read.*` | Peut lire tous les documents |
| `*.*.*` | Accès complet (admin bypass) |

### Wildcards

Le caractère `*` fonctionne comme wildcard et peut apparaître à n'importe quelle position :

- `users.*` — toutes les actions sur le module `users`
- `*.read` — action `read` sur tous les modules
- `*.*.*` ou `*` — accès complet à tout

### Granularité

Le troisième niveau (`granularité`) définit **la portée** de la permission. Les valeurs courantes sont :

| Granularité | Description |
|-------------|-------------|
| `own` | Uniquement les ressources créées par l'utilisateur |
| `department` | Les ressources du département de l'utilisateur |
| `organization` | Les ressources de l'organisation de l'utilisateur |
| `*` | Toutes les ressources, sans restriction |

Si la granularité est omise, on suppose `*` (sans restriction).

## Attribution des rôles avec Casdoor

**Casdoor** gère le mappage entre utilisateurs, rôles et permissions. Le flux est le suivant :

1. Un **utilisateur** est attribué à un ou plusieurs **rôles** dans Casdoor
2. Chaque **rôle** a un ensemble de **permissions** associées
3. Lorsque l'utilisateur s'authentifie, le JWT contient les rôles attribués
4. Le backend extrait les permissions des rôles et vérifie l'accès

### Structure dans Casdoor

```text
Utilisateur : m.rossi@example.com
  └── Rôles : [admin, editor]
        ├── admin → [*.*.*]
        └── editor → [documents.read.*, documents.write.own, users.read]
```

### Jeton JWT

Le JWT émis par Casdoor inclut les rôles de l'utilisateur dans le claim correspondant. Le backend
résout les rôles dans les permissions correspondantes et les stocke dans le contexte de la requête
pour les vérifications d'autorisation.

## Vérification des permissions dans le backend

Le backend vérifie les permissions à chaque requête. La logique de vérification compare la permission
requise par l'endpoint avec les permissions effectives de l'utilisateur :

```typescript
// Exemple conceptuel de la vérification
function hasPermission(userPermissions: string[], required: string): boolean {
  return userPermissions.some(p => matchWildcard(p, required));
}

// Vérifie si "users.read.*" couvre "users.read.own"
hasPermission(['users.read.*'], 'users.read.own'); // true

// Vérifie si "*.read" couvre "documents.read"
hasPermission(['*.read'], 'documents.read'); // true
```

Le matching des wildcards suit des règles standards : `*` à une position correspond à
n'importe quelle valeur à cette position.

## Admin bypass

Les utilisateurs avec le rôle **admin** (ou avec la permission `*.*.*`) ont un **accès complet**
à toutes les ressources et toutes les actions, contournant toutes les vérifications d'autorisation.

Cela est implémenté comme première vérification dans le middleware d'autorisation :

```text
SI l'utilisateur a la permission *.*.* :
  → permet l'accès (bypass complet)
SINON :
  → vérifie la permission spécifique requise par l'endpoint
```

## Endpoints et permissions requises

Chaque endpoint du backend déclare la permission requise. Par exemple :

| Endpoint | Permission requise |
|----------|---------------------|
| `GET /api/v1/users` | `users.read` |
| `POST /api/v1/users` | `users.write` |
| `DELETE /api/v1/users/:id` | `users.delete` |
| `GET /api/v1/ws/document-service/documents` | `documents.read` |

Si l'utilisateur n'a pas la permission requise, le backend renvoie une erreur **403 Forbidden**
avec un Problem Details indiquant la permission manquante (voir [Gestion des erreurs](/fr/api/error-handling/)).
