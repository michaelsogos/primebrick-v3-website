# Introduction à l'API

L'API REST de Primebrick est l'interface principale pour interagir avec le système. Toutes les
requêtes passent par le backend Express, qui sert de gateway unique pour le frontend, les
microservices et les clients externes.

## URL de base

L'URL de base de l'API dépend de l'environnement :

| Environnement | URL de base |
|---------------|-------------|
| Développement local | `http://localhost:3001/api/v1` |
| Staging | `https://api.staging.primebrick.dev/api/v1` |
| Production | `https://api.primebrick.dev/api/v1` |

Tous les endpoints sont servis sous le préfixe **`/api/v1/`**. La version de l'API fait partie de
l'URL, garantissant que les changements non rétrocompatibles soient introduits sous un nouveau préfixe
(ex. `/api/v2/`) sans interrompre les clients existants.

## Structure des endpoints

### Endpoints du backend

Les endpoints natifs du backend suivent la structure :

```text
GET    /api/v1/users              — Liste des utilisateurs
POST   /api/v1/users              — Créer un utilisateur
GET    /api/v1/users/:id          — Détail d'un utilisateur
PUT    /api/v1/users/:id          — Mettre à jour un utilisateur
DELETE /api/v1/users/:id          — Supprimer un utilisateur
```

### Endpoints des microservices

Les endpoints exposés par les microservices sont accessibles via le proxy du backend, sous
le préfixe **`/ws/:serviceCode/`** :

```text
GET    /api/v1/ws/document-service/documents       — Liste des documents
POST   /api/v1/ws/document-service/documents       — Créer un document
GET    /api/v1/ws/document-service/documents/:id   — Détail d'un document
```

Le `serviceCode` identifie le microservice. Le backend route la requête vers le microservice
correspondant avec équilibrage round-robin s'il y a plusieurs instances.

## Endpoints publics

Certains endpoints ne nécessitent pas d'authentification :

| Endpoint | Description |
|----------|-------------|
| `GET /api/v1/health` | Health check du backend |
| `GET /api/v1/openapi.json` | Spécification OpenAPI du backend |
| `GET /api/v1/openapi/aggregated.json` | Spécification OpenAPI agrégée (backend + microservices en ligne) |

L'endpoint OpenAPI agrégé est public pour permettre à l'API Explorer de fonctionner sans authentification.

## Conventions

### Format des réponses

Les réponses réussies renvoient le payload directement au format JSON :

```json
{
  "id": "usr_abc123",
  "username": "mrossi",
  "email": "m.rossi@example.com",
  "createdAt": "2025-01-15T10:30:00Z"
}
```

Les réponses d'erreur suivent le standard **RFC 7807 Problem Details** (voir
[Gestion des erreurs](/fr/api/error-handling/)).

### Pagination

Les endpoints qui renvoient des listes prennent en charge la pagination via les query parameters `page` et
`pageSize` :

```text
GET /api/v1/users?page=2&pageSize=20
```

La réponse inclut les métadonnées de pagination :

```json
{
  "data": [...],
  "pagination": {
    "page": 2,
    "pageSize": 20,
    "totalItems": 150,
    "totalPages": 8
  }
}
```

### Filtres et tri

De nombreux endpoints prennent en charge les filtres et le tri via des query parameters :

```text
GET /api/v1/users?role=admin&sortBy=createdAt&sortOrder=desc
```

## API Explorer

Primebrick inclut un **API Explorer** interactif basé sur Scalar, accessible depuis la
page [API Explorer](/api-explorer/). L'explorer :

- Charge la spécification OpenAPI agrégée côté client
- Permet de configurer l'URL du backend (dev, staging, production)
- Prend en charge l'authentification avec bearer token, API key ou OAuth2
- Permet de tester les endpoints directement depuis le navigateur
